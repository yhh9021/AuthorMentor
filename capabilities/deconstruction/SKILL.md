# 拆书能力

## 目标

把输入材料拆解为可复用的网文生产知识，服务全局素材库或单书专属素材库。拆书不是文学评论，也不是复述剧情。

长篇小说必须使用长程分段拆书：先建立全书画像和设定库，再按剧情阶段或默认 20 章左右的区间逐段精拆，最后抽取关键事件链、高光片段、优点机制，并做产物有效性审计。

如果不能区分本书类型、设定、人物、势力和关键亮点，不允许输出“已完成”。

如果用户要求“深入拆”“全书分析”“人物关系图”“完整设定集”“每一项都基于全本内容”，优先使用 `author-mentor deconstruct deep-run <source>`，不要只跑普通 `full-run`。

如果用户要求持续 review 每轮结果、判断产物是否足够有深度、可参考、可复用，必须先读 `references/depth-quality-standard.md`，并把审计结果作为下一轮 Prompt、Skill 或 runner 优化依据。

`deep-run` 支持子 Agent 精读：

- 默认 `--agent-mode fallback`：生成 chunk 精读任务包，读取已有子 Agent JSON；缺失时用脚本抽取兜底。
- 严格 `--agent-mode required`：要求每个 chunk 都有子 Agent JSON，否则失败。
- 可用 `--agent-command "<command>"` 指定外部 Agent 编排命令。命令会收到 `AUTHOR_MENTOR_AGENT_INPUT_DIR` 和 `AUTHOR_MENTOR_AGENT_OUTPUT_DIR` 两个环境变量。
- chunk 正文任务包写入 `runs/<run-id>/input/agent-chunks/`，该目录被 Git 忽略，不提交原文。
- 子 Agent 只需要把同名 JSON 写入 `runs/<run-id>/output/agent-chunks/`，最终合并器会读取这些 JSON。

## 每轮 review 闭环

生成或更新拆书产物后，必须执行产物有效性审计：

```bash
pnpm dev deconstruct audit <单书目录或global/deconstructions>
```

需要作为硬门禁时使用：

```bash
pnpm dev deconstruct audit <目录> --fail-on-issues
```

审计报告会写入单书目录的 `产物有效性审计.md`，或全局拆书目录的 `产物有效性总审计.md`。

如果审计结果为“需返工”，不能只润色 Markdown。必须判断问题属于哪一类：

- Prompt/Skill 约束太弱：先修改 Skill 或参考文件。
- runner 生成逻辑模板化：先修改 TypeScript 生成逻辑或增加结构化输入。
- 子 Agent JSON 覆盖不足：补跑 chunk 子 Agent 精读。
- 当前产物未重跑：按修正后的能力重跑。

只有在审计没有中高严重问题，且产物能回答“可复用什么、为什么好、如何改写、不能复用什么”后，才允许声称本轮产物有效。

`full-run` 只能作为初拆索引；当审计指出“模板句重复、机制只有标题级列表、设定缺少运行规则或人物接口”时，必须升级为 `deep-run` 或子 Agent 精读返工。

审计失败后的子 Agent 返工流程：

```bash
pnpm dev deconstruct refine-prepare <单书目录>
```

然后启动子 Agent 读取任务包 `TASK.md`、`input/` 和原文。新版返工优先使用专题子 Agent：每个专题只编辑 `output/topics/` 下自己的 JSON，不共享上下文、不互相复写内容。

专题子 Agent 分工：

- `plot.json`：剧情阶段和关键事件链。
- `characters.json`：人物画像、身份变化和语义关系边。
- `factions.json`：阵营、集团、政权、家族、军政组织和势力范围。
- `map.json`：地点、区域、空间层级、通道和空间内势力分布。
- `settings.json`：世界规则、能力体系、资源体系和禁忌代价。
- `highlights.json`：高光片段和为什么好看的机制。

旧的 `output/refine-insights.json` 只作为兼容事实包，不再作为所有专业产物的唯一内容来源。子 Agent 必须产出对应专题 JSON，不允许只润色旧报告。

子 Agent 填写完成后应用返工产物：

```bash
pnpm dev deconstruct refine-apply <runs下的返工任务包目录>
pnpm dev deconstruct audit <单书目录>
```

`refine-apply` 会更新单书目录中的 `全书拆书总报告.md`、`剧情阶段总览.md`、`关键事件链.md`、`深度高光片段.md`、`深度设定沉淀.md`、`优点与可复用机制.md`、`人物与关系图.md`、`修炼能力与资源体系.md`、`设定集-*.md` 和 `深拆中间数据.json`，并提交 Git 记录。

脚本、程序和 JSON 的职责边界：

- 脚本负责流程编排、任务拆分、文件落盘、schema 校验、确定性审计和 Git 记录。
- JSON 是子 Agent 之间交接的结构化协议，不是内容理解来源。
- 只要涉及内容理解和判断，例如“哪些人物重要”“势力范围如何变化”“哪个桥段是高光”“设定应归哪一类”，必须由专题子 Agent 完成。
- renderer 只做格式化，不用关键词把同一批内容硬塞进不同产物。

返工人物关系时必须执行角色召回校验：

- `refine-prepare` 会从原文生成 `input/角色召回候选.json`。
- 子 Agent 必须先读这个文件，再写 `characterNetwork.profiles` 和 `characterNetwork.relations`。
- `mustReview=true` 的候选表示原文高频命中的高价值角色，必须进入人物画像和关系边；如果判断不是剧情人物，必须在 `characterNetwork.misreads` 写清排除理由。
- 历史/三国类作品要额外核对姓名、字、别名和同姓不同人，例如 `赵云/赵子龙/子龙` 不能因为人物数量已达标而漏掉。
- 审计会检查高价值角色召回缺失；数量达标但关键人物缺失，仍视为需返工。

势力与组织必须由 `factions.json` 生成：

- 它不是制度规则库，也不是资源体系库。
- 条目必须是有成员、地盘、资源、敌我关系和阶段变化的阵营/集团/政权/家族/军政组织。
- 历史题材应类似“公孙珣势力、袁绍势力、曹操势力、董卓势力、汉室朝廷、黄巾势力、鲜卑/乌桓势力”等。
- “经学、军功、婚姻、水利、征辟、官职秩级”这类制度规则不得进入 `设定集-势力与组织.md`。

## 输入

- 原始小说文本：用户提供的小说正文文件。
- 二手拆书来源：公开书评总结或拆书视频整理文本。

普通读者评论不能作为拆书内容，但可以作为二手来源可信度的互动信号。

## 拆解维度

### 全书结构

- 类型画像
- 开局设计
- 核心卖点
- 主角成长线
- 金手指或能力系统
- 冲突升级
- 地图或势力展开
- 章节钩子
- 爽点兑现
- 读者期待管理
- 可复用模式
- 复用风险

### 设定集沉淀

必须按类型拆出可检索的 Story Bible/Codex 条目：

- 世界规则：基本世界观、超凡/科技/历史规则、禁忌、代价。
- 能力体系：境界、序列、职业、技能、科技树、评价标准。
- 资源体系：钱、灵石、材料、知识、权限、社会资本、信息差。
- 组织势力：宗门、学院、教会、公司、国家、军队、秘密组织、家族。
- 地图层级：村镇、城市、国家、异界、宇宙、秘境、战场。
- 人物关系：主角、导师、同伴、对手、反派、交易对象、上位者。
- 时间线：必须使用小说世界内部的年号、纪元、朝代年份或卷内年份，记录大事件、伏笔埋设、伏笔升级、伏笔回收；章节号只能作为证据定位，不能替代时间。
- 写作规则：本书独特叙事口味、爽点兑现方式、禁用风险。

每条设定必须写成可复用的完整条目，不能只写一两句话。至少包含：定义、运行规则、代价或限制、人物/势力接口、阶段变化、为什么值得借鉴、复用边界。每条设定尽量记录：首次出现章节、升级章节、回收章节、功能、复用边界。

### 独立设定集

每本书必须有独立设定集，作为后续自动写作的事实库。设定集不是普通总结，必须能被章节生成和审核流程检索使用。

设定集至少覆盖：

- 总览：题材承诺、核心世界观、设定如何服务爽点。
- 修炼/能力/资源体系：必须按题材适配。玄幻、仙侠、高武、西幻、科幻可拆境界、序列、职业、功法、魔法、武道、科技树、升级路径、能力代价；历史、官场、战争等无超凡修炼题材不能硬套“修炼”，应拆军政能力、官职制度、兵粮财政、人脉名望、门生故吏、地缘交通、正统叙事和信息差。
- 地图与空间层级：必须覆盖全书主要地图、空间层级、通道、边界和势力分布。历史题材先识别朝代/时代，再结合模型内置知识或网络搜索补全真实地理与势力格局，最后回到原文标注本书改写；架空、仙侠、科幻题材必须依赖全文总结，不得只列少数地点。
- 势力与组织：宗门、学院、教会、朝廷、公司、军队、秘密组织、家族、敌对阵营。
- 资源体系：钱、灵石、材料、知识、权限、社会资本、信息差、装备、丹药、军政声望。
- 人物关系与身份体系：只收人物，不收规则、组织、地点或抽象设定。必须区分主角、核心人物、重要人物、次要人物；主角和核心人物需要更详尽的身份弧线、关系变化、关键矛盾和调用注意，次要人物可以更简略。
- 世界规则与禁忌：污染、失控、神灵、历史规则、制度约束、不可触碰的代价。
- 设定时间线：用小说内时间排序关键设定首次出现、升级、反转和回收；不能用“第几章”当时间线标题。
- 写作复用边界：可复用结构、不可迁移专有名词和不可迁移事件组合。

每个设定集条目至少包含：设定定义、运行规则、剧情落点、代价限制、人物/势力接口、阶段变化、复用边界。

### 全本结构化深拆

深拆不能只凭章节标题、开头片段或预设画像推断。必须先把全本正文按 chunk 扫描，生成结构化中间数据，再合并为报告。

深拆中间数据至少包含：

- 人物：姓名/别名候选、首次出现章节、最后出现章节、活跃章节数、身份线索。
- 关系边：人物 A、人物 B、章节范围、关系类型、关系证据；不能只输出共现次数。
- 人物功能分层：主角候选、主要对手/反派、核心同伴/盟友、导师/上位者/下属、亲属/情感、交易/利用、重要配角。
- 主角关系网：主角与关键人物之间的关系类型、矛盾点、合作点、身份牵引和剧情功能。
- 势力组织：组织名、首次出现章节、章节范围、关联人物线索。
- 地图地点：地点名、首次出现章节、章节范围、空间层级线索。
- 修炼/能力/资源规则：按题材抽取体系名、资源名、规则名、章节范围、功能线索。若本书无修炼体系，必须明确写“本书无修炼体系”，并转为现实能力与资源拆解；禁止把“时期、首级、人名、情境”等正则噪声当成体系。
- chunk 摘要：每个 chunk 命中的人物、势力、地点、体系。

子 Agent JSON 必须包含：

- `characters`：人物名、身份/功能、证据摘要、关系线索。
- `relations`：人物 A、人物 B、关系类型、关系方向或互动性质、证据摘要。关系类型必须尽量落到“敌对/矛盾、同盟/合作、师徒/上下级、亲属/情感、交易/利用、组织同盟、普通共现”之一；不能只写“有关联”。
- `organizations`：势力/组织名、组织功能、证据摘要。
- `locations`：地点/地图名、空间层级或功能、证据摘要。
- `systems`：能力/资源/规则名、类别、运行规则、证据摘要。历史类 `systems` 应优先输出官职、军队、兵粮、财政、名望、人脉、地缘、正统叙事、信息差等现实资源规则，不要编造境界或修炼。
- `events`：关键事件、证据摘要、后续影响。
- `timeline`：小说内时间、章节定位、事件、涉及人物/势力、当时世界局势、后续影响、证据摘要。时间字段必须是年号/纪元/朝代年份/卷内年份，不得写章节号。
- `storyEvents` 或返工 `events`：比高光更细的事件链条，优先按故事真实进展切分；长篇默认接近 5-12 章一个事件，不能只复用高光列表。
- 返工 `locations`：地点/区域名、类型层级、上级区域、控制者/势力、剧情功能、连接关系、势力分布、章节证据。
- `highlights`：本 chunk 内值得沉淀的高光片段，必须包含剧情概述、前置铺垫、冲突设计、当场兑现、后续影响、可复用机制、复用边界和证据摘要。
- `settingInsights`：本 chunk 内可进入设定集的规则洞察，必须包含设定定义、运行规则、代价限制、人物/势力接口、阶段变化、借鉴价值、复用边界和证据摘要。
- `mechanisms`：本 chunk 内“为什么好看”的机制，必须包含机制原理、本书实现、读者吸引力、改写方法、失败风险和证据摘要。

产物有效性审计必须检查：

- 人物数量是否达到与章节体量匹配的最低阈值。
- 关系边是否有足够语义类型支撑人物关系图，是否能回答“主角是谁、谁是反派/对手、谁是同伴/盟友、谁和谁有矛盾、谁和谁有身份或利益联系”。
- 势力、地图、能力/资源/规则是否都有全书级候选表。
- `设定集-设定时间线.md` 是否使用小说内年份/纪元，而不是章节号。
- `设定集-人物关系与身份体系.md` 是否混入非人物条目，且是否区分主次详略。
- `剧情阶段总览.md` 与 `关键事件链.md` 是否粒度不同：总览应粗，事件链应细。
- `设定集-地图与空间层级.md` 是否覆盖主要地图和势力分布，而不是只列少数地点。
- 每个 chunk 是否至少沉淀一类结构化信息。
- 输出是否大量模板句；如果是，必须回到 chunk 重新抽取。
- 高光、设定和机制是否来自子 Agent 精读 JSON；如果仍来自 `full-run` 的模板化初拆，视为需返工。

深拆失败时不能只润色 Markdown，必须回到中间 JSON 或 chunk 抽取步骤返工。

### 分段精拆

每个剧情段都要尽量记录：

- 章节范围
- 阶段目标
- 主要事件链
- 新增设定
- 人物关系变化
- 冲突设计
- 爽点设计
- 钩子与回收
- 节奏变化
- 可复用桥段
- 不可复用风险

### 高光片段

高光不是“章节标题列表”。高光数量按书的体量和讨论度动态确定：

- 500 章以下：至少 8 个。
- 500-849 章：至少 10 个。
- 850-1199 章：至少 12 个。
- 1200 章以上：至少 16 个。

如果是 1000 万字级作品，或网上讨论明显集中在多个长篇阶段，需要继续增加，不得机械封顶。

高光候选必须结合正文、章节索引和网上讨论信号。优先搜索“书名 + 高光/名场面/书评/为什么好看/讨论度高”等公开内容，再回到本地正文定位章节范围。

每个高光片段必须记录：

- 章节范围
- 剧情概述：这一段大概发生了什么，不能只列章节标题。
- 为什么高光：新设定、新能力、新关系、新地图、情绪爆点、战斗爽点、反转、伏笔回收。
- 前置铺垫
- 当场兑现
- 后续影响
- 可复用机制
- 不可复用元素

每个高光条目至少应有 6 个分析点；如果只能写出一两句话，说明没有真正拆开，必须回到正文重读。

每本书必须沉淀 5-10 条“为什么好看”的机制，每条机制要写清楚原理、本书如何实现、为什么好看、适用题材、改写方法和失败风险。例如：

如果初始画像里可复用机制不足 5 条，必须从本书核心优点、高光维度和分段精拆结论中补足，不能按不足数量直接输出。

- 金手指如何分阶段解锁。
- 资源压力如何可视化。
- 主角如何获得信息差。
- 公开赛道如何制造打脸和排名爽点。
- 生产技能如何变成长期收益。
- 单元案件如何串成长线阴谋。
- 组织身份如何升级。
- 早期伏笔如何抬升为终局世界观。

### 关键事件链与复用素材

可复用素材不再单独沉淀为素材卡文件，必须合并进 `关键事件链.md`、`深度高光片段.md`、`优点与可复用机制.md` 和设定集。重点记录可以被未来创作复用的局部结构，包括：

- 设定模式
- 剧情桥段
- 爽点设计
- 章节钩子
- 伏笔与回收
- 人物关系
- 反派或压力源
- 资源与交易
- 战斗或斗法
- 节奏结构
- 梗与趣味点
- 风险与禁用项

每个可复用素材必须有明确来源章节和复用边界；不能只写“本段承担某某功能”。

`剧情阶段总览.md` 与 `关键事件链.md` 必须分工：

- `剧情阶段总览.md`：粗粒度，通常 4-8 个阶段，回答全书主线如何推进。
- `关键事件链.md`：细粒度，按故事真实进展切分，优先 5-12 章一个事件或一个完整因果单元，记录前因、冲突升级、关键转折、当场结果、后续影响和涉及人物/势力。
- 如果暂时无法理解全文自然切分，才允许按 10 章左右兜底；不能直接复制高光片段列表。

## 工作流

1. 读取正文，识别编码、章节标题格式、章节数和可能的卷/部结构。
2. 建立本书类型画像，禁止把其他书的题材标签套入本书。
3. 先生成 `章节索引.md` 和 `剧情阶段总览.md`。
4. 生成 `深度设定沉淀.md` 和 `设定集-*.md`，作为后续分段拆解和章节写作的事实源。
5. 分段精拆，每段都要更新事件、设定、关系、冲突、爽点、钩子。
6. 抽取高光片段、细粒度关键事件链和优点机制，不够具体时必须回到正文重读。
7. 把可复用素材合并进关键事件链、深度高光、优点机制、设定集和全局/单书素材库；同时生成小说内年份时间线、人物主次分层和地图势力分布。
8. 做产物有效性审计，检查跨书污染、题材错配、章节数异常、空洞模板化。
9. 跑 `author-mentor deconstruct audit`，审计产物是否足够深入、可参考、可复用。
10. 审计未通过时，先优化 Prompt/Skill/runner，再回到正文或 chunk 精读返工，直到审计通过或明确剩余问题需要新的外部输入。

## 产物有效性审计

输出前必须检查：

- 章节数是否合理，是否只解析到 1 章或漏掉卷内编号。
- 目录和报告中的书名是否一致。
- 非仙侠作品中是否出现明显仙侠专属标签，例如“道院、水府、制符、筑基、元婴、魔主”。
- 非西幻作品中是否出现“奥术、教会、魔法论文”等错配标签。
- 非历史作品中是否出现“州郡、汉室、诸侯”等错配标签。
- 总报告是否与其它书重复超过 30%。
- 每本书是否至少有设定集、优点机制、关键事件链、高光片段和产物有效性审计。
- 深度设定、高光和优点机制是否存在大量“一句话条目”；如果存在，视为失败。
- 设定时间线是否用年号/纪元而不是章节号。
- 人物身份体系是否混入非人物条目，且主次人物详略是否区分。
- 地图空间层级是否覆盖主要地理、通道和势力分布。
- 势力与组织是否按势力范围、核心人物、控制区域、资源兵力、盟友敌对和阶段变化整理，而不是混入制度规则。
- 剧情总览和关键事件链是否只是同一批高光的重复表达。
- 不同产物之间是否复用大段相同内容；跨文件重复视为专题子 Agent 分工失败。

如果发现问题，必须先修正画像、解析或分段，再重新生成。

## 复用边界

只能提取结构、节奏和功能。禁止迁移来源作品中的独特表达、专有名称、独特事件链或可识别组合。

## 二手来源可信度

如果输入是二手拆书来源，需要说明可信度依据，至少考虑：

- 来源本身点赞数或互动量
- 高赞评论反馈
- 创作者历史质量
- 多个来源是否互相印证

## 外部方案启发

需要调研或扩展能力时，先读 `references/外部方案调研.md`。关键结论：

- 像 NovelCrafter Codex / AI Dungeon Story Cards / SillyTavern World Info 这类系统，都把角色、地点、物品、组织、规则等拆成可检索的记忆条目，而不是塞进单一总结。
- 像 autonovel 这类自动小说 pipeline，会先做 world、characters、outline、voice、canon，再写章节，并通过评审循环迭代。
- 因此拆书产物必须服务后续写作：设定库、剧情阶段、角色/势力关系、高光机制和产物有效性审计都要结构化。
