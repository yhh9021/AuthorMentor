# 深度质量标准

本标准用于判断拆书产物是否能服务后续自动写作。拆书不是“看起来分析过”，而是要沉淀可检索、可复用、可校验的创作资产。

## 合格产物的定义

合格产物必须同时满足：

- 有证据：关键结论能追溯到章节范围、事件链、人物关系或子 Agent 精读 JSON。
- 有细节：不是一两句话概括，而是写清发生了什么、为什么有效、如何复用、哪里不能复用。
- 有差异：不同设定、不同高光、不同机制不能套同一段模板话。
- 有接口：设定要说明人物/势力/资源/地图如何连接；机制要说明它如何进入章节生成。
- 有边界：明确哪些结构可复用，哪些专名、关系组合和事件链不能迁移。
- 可返工：审计发现问题时，必须回到正文、chunk JSON 或子 Agent 精读步骤，不允许只润色 Markdown。

## 高光片段标准

每个高光条目必须包含：

- 章节范围。
- 剧情概述：用具体事件说明这一段发生了什么。
- 剧情拆分：起点、推进、结果，至少说明三段因果。
- 前置铺垫：这个爽点或反转之前埋了什么。
- 冲突设计：主角面对的限制、对手、代价或信息差。
- 当场兑现：读者当场获得的爽点、情绪、真相、资源或身份变化。
- 后续影响：它如何改变之后的人物关系、地图、能力体系、势力格局或终局伏笔。
- 为什么好：不能只说“推进剧情”，要说明它为什么让读者想追。
- 可复用写法：抽象成可迁移的功能链。
- 不可复用元素：明确禁止迁移的专名、事件组合和标志性表达。

失败特征：

- 只列章节标题。
- 多个高光使用同一套“前置压力 -> 规则限制 -> 当场兑现”空泛话术。
- 没写具体角色、势力、资源、能力、地图或矛盾。
- 只说“后续影响深远”，不说明影响到哪里。

## 设定集标准

每个设定条目必须包含：

- 设定定义：这个设定是什么，不是什么。
- 运行规则：谁能使用、怎样升级、如何结算、失败后果是什么。
- 剧情落点：首次出现、升级、反转、回收的章节范围。
- 代价限制：资源成本、身份风险、组织约束、污染/失控/制度限制等。
- 人物/势力接口：谁解释规则、谁垄断规则、谁因主角突破规则受益或受损。
- 阶段变化：前期、中期、后期功能如何变化。
- 写作用途：后续章节生成时应如何查询和遵守。
- 复用边界：可复用结构和不可迁移内容。

失败特征：

- 所有设定条目都用同一套“规则分层、成本递增、人物接口、阶段升级”解释。
- 写成世界观名词表，没有运行规则。
- 没有代价、限制和人物接口。
- 只说“需要二次精读补充”。

## 优点机制标准

每条机制必须写清：

- 机制原理。
- 本书具体如何实现。
- 为什么好看。
- 适用题材。
- 改写方法。
- 失败风险。
- 来源章节或高光片段。

失败特征：

- 只有“核心优点”和“可复用机制”列表。
- 机制之间只是换名词，解释完全一样。
- 没有说明如何改写成新书章节。

## 人物关系标准

人物关系图必须回答：

- 主角是谁。
- 谁是核心同伴/盟友。
- 谁是主要对手/反派。
- 谁是导师/上位者/下属。
- 谁有亲属、情感或特殊羁绊。
- 谁和谁有矛盾、合作、交易、组织身份或阶段变化。
- 自动抽取中哪些节点和关系明显误判。

只统计共现次数不合格。

## 每轮审计流程

每次生成或更新拆书产物后，必须执行：

```bash
pnpm dev deconstruct audit <单书目录或global/deconstructions>
```

如果需要作为门禁：

```bash
pnpm dev deconstruct audit <目录> --fail-on-issues
```

审计结果为“需返工”时：

1. 先读 `产物有效性审计.md` 或 `产物有效性总审计.md`。
2. 判断问题来自 Prompt、Skill、runner、子 Agent JSON 覆盖率，还是当前产物未重跑。
3. 优先修 Prompt/Skill/runner，再补跑产物。
4. 如果问题是“模板化、标题级、人物关系机械统计、没有全本证据”，必须运行 `pnpm dev deconstruct refine-prepare <单书目录>` 生成返工任务包，并让子 Agent 读取原文填写 `output/refine-insights.json`。
5. 子 Agent 返工输出必须同时覆盖高光、设定、机制、人物画像和语义关系边；完成后运行 `pnpm dev deconstruct refine-apply <返工任务包目录>`。
6. 重新审计，直到没有中高严重问题，或明确剩余问题必须依赖新的原文精读输入。
