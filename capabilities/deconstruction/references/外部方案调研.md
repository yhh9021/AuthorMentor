# 外部方案调研

## 调研结论

拆书能力不能只输出剧情摘要。成熟写作工具和自动小说 pipeline 都把“可持续写作”拆成结构化资产：世界、角色、地点、物品、组织、规则、剧情线、风格、canon 和评审记录。

## 参考方案

### NovelCrafter Codex

- 官方说明把 Codex 定位为存放角色、地点、物品等核心故事信息的中心。
- Codex 类型包括 character、location、object、lore、subplot、other。
- 对我们的启发：拆书输出应形成 Story Bible 条目，不应只写一篇总报告。

参考：

- https://www.novelcrafter.com/help/docs/codex/the-codex
- https://www.novelcrafter.com/help/docs/codex/codex-types
- https://www.novelcrafter.com/features/codex

### SillyTavern World Info / Lorebook

- World Info 通过关键词动态插入相关世界观信息。
- 每个 entry 需要独立、简洁、可被触发，支持优先级、位置和上下文预算。
- 对我们的启发：设定沉淀要拆成可检索条目，记录触发关键词、适用范围、复用边界。

参考：

- https://docs.sillytavern.app/usage/core-concepts/worldinfo/

### NousResearch autonovel

- autonovel 是自动小说生产 pipeline。
- Foundation 阶段先生成 world、characters、outline、voice、canon，再进入章节写作。
- 文件结构明确包含 `world.md`、`characters.md`、`outline.md`、`canon.md`、`voice.md`。
- 评审有机械检查和 LLM judge，还包含后续 review loop。
- 对我们的启发：拆书也应反向抽取这些层：世界、角色、剧情、canon、风格、质量审计。

参考：

- https://github.com/NousResearch/autonovel

## 转化为本项目要求

每本书拆完至少要沉淀：

- `synthesis/设定沉淀.md`
- `synthesis/优点与可复用机制.md`
- `synthesis/高光片段与亮点.md`
- `synthesis/质量审计.md`
- `segments/` 分段精拆
- `material-cards/` 细粒度素材卡

质量审计必须检查跨书污染和题材错配。比如《奥术神座》不应出现“道院、水府、制符”，《覆汉》不应被拆成修仙升级，《诡秘之主》不应被拆成生产技能制符。
